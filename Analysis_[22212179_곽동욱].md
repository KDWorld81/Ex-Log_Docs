# 1. Introduction

## 1.1 Summary

Ex-Log는 운동을 게임처럼 즐길 수 있도록 설계된 운동 기록 서비스입니다. 운동을 해서 경험치를 받아서 쌓고 받은 경험치로 티어를 올리고, 칭호를 모으는 방식으로 운동 습관을 만들어가며 동기부여를 줍니다

기획의 출발점은 동기부여입니다. 헬스장을 자신의 돈을 내고 일주일 만에 혹은 한달만에 가지않는 경험도 주변인들중에 굉장히 많습니다.
그리고 운동 앱을 깔고 기록을 남기다가 어느 순간 열지 않게 되는 경험. 한국 청소년의 94.4%가 신체 활동 부족 상태라는 통계는 이 현실을 숫자로 보여줍니다.
문제는 의지가 아닙니다. 운동을 계속하게 만드는 서비스가 없는것 같았습니다.

직장인들이 운동을 못 하는 이유로 시간 부족을 꼽지만, 실제로는 다릅니다. 오늘 스쿼트 50개를 했는데 그게 지난주보다 나아졌는지 알 수 없고, 하루 빠졌을 때 눈에 보이는 손해가 없으며, 혼자 하다 보면 성취감보다 막막함이 먼저 옵니다. 운동이 고통스럽고 의무적인 행위로 굳어지는 건 이 과정에서 자연스럽게 일어납니다.

Ex-Log는 운동이라는 행위를 바꾸려 하지 않습니다. 운동을 둘러싼 경험의 구조를 바꿉니다. 스쿼트 50개를 하면 경험치가 올라가고, 30일을 꾸준히 채우면 잔디가 빼곡해지고, 하체 운동 100세트를 달성하면 칭호가 생깁니다. 운동이 숫자로 남고, 그 숫자가 성장으로 이어지는 과정이 눈에 보일 때 사람은 같은 행동을 스스로 반복하게 됩니다.

게임이 수십 년간 사람들을 붙잡아온 원리가 여기 있습니다. 즉각적인 피드백, 성장의 시각화, 다음 목표의 선명함. Ex-Log는 이 원리를 건강한 행동에 그대로 가져옵니다.

## 1.2 Project Description

### 🎮 경험치 및 티어 알고리즘

경험치 계산은 운동 종류, 강도, 빈도를 함께 반영합니다. 단순히 오래 하거나 무겁게 들수록 많이 주는 방식이 아닙니다. 그렇게 설계하면 벤치프레스만 매일 반복하는 편향된 패턴이 생깁니다.

Ex-Log의 ExpService는 하루에 3종목 이상을 수행했을 때 보너스 경험치를 부여합니다. 스쿼트만 100세트 한 사람보다 스쿼트, 벤치프레스, 달리기를 함께 한 사람이 더 높은 경험치를 얻습니다. 새로운 운동에 도전해야 할 이유를 억지로 만들지 않아도, 보너스를 얻고 싶다는 자연스러운 동기가 생깁니다.

티어는 브론즈, 실버, 골드, 플래티넘, 다이아몬드 순으로 올라갑니다. 각 단계를 올라갈수록 필요한 경험치 임계값이 높아집니다. 현재 티어에서 다음 티어까지 남은 경험치가 얼마인지 보이기 때문에, 목표가 추상적이지 않습니다. 오늘 운동을 하면 그 거리가 얼마나 줄어드는지 바로 확인할 수 있습니다.

### 🌱 활동 가시화

연간 365칸의 그리드에 운동한 날마다 색이 채워집니다. 가볍게 스트레칭만 한 날은 연하게, 여러 종목을 고강도로 수행한 날은 진하게 표시됩니다. 출석 여부만 체크하는 게 아니라 그날 운동의 밀도가 색에 담기는 구조입니다.

GitHub 잔디를 며칠 채우다 보면 끊기 싫다는 감각이 생깁니다. 심리학에서는 이를 연속성 유지 동기라고 부르는데, 자신이 만들어온 패턴을 깨는 것에 본능적인 저항이 생기는 현상입니다. 오늘 운동을 건너뛰는 것이 단순한 게으름이 아니라 내가 쌓아온 기록에 구멍을 내는 행위로 느껴지기 시작합니다.

활동 맵은 프로필에 공개됩니다. 빽빽하게 채워진 잔디는 말보다 강한 증거입니다.

### 🏆 칭호 및 도감 시스템

칭호는 특정 조건을 채울 때 수여됩니다. 누적 하체 운동 100세트를 채우면 하체의 왕, 30일 연속 출석을 유지하면 불굴의 의지, 하루에 5종목 이상을 수행하면 멀티플레이어 칭호를 얻습니다. 많이 하는 것에만 보상이 있는 게 아닙니다. 꾸준히 한 것, 다양하게 한 것에도 각각의 칭호가 주어집니다.

프로필에 붙는 칭호는 숫자로 표현할 수 없는 맥락을 담습니다. 같은 골드 티어라도 하체 특화 칭호를 단 사람과 전신 균형 칭호를 단 사람의 운동 히스토리는 전혀 다릅니다.

도감은 아직 얻지 못한 칭호와 획득 조건을 보여줍니다. 사용자는 도감을 보면서 지금 어떤 조건에 가장 가까운지, 어떤 운동을 추가하면 원하는 칭호를 얻을 수 있는지 스스로 계산합니다. 외부에서 부여된 미션이 아니라 본인이 직접 찾아낸 목표이기 때문에 동기의 강도가 다릅니다.

### 🛡️ 백엔드 아키텍처 및 보안

Spring Boot 와 JPA

controller, service, repository, entity, dto로 패키지 혹인 domain 별로 계층분리를 명확하게하여 구현했습니다.
각 레이어가 자신의 역할에만 집중하고 다른 레이어의 구현 방식에 의존하지 않습니다. 
경험치 계산 로직을 수정할 때 컨트롤러 코드를 손댈 필요가 없고, API 응답 형식을 바꿀 때 서비스 로직에 영향을 주지가 않습니다.

JPA를 통해 운동 데이터와 경험치 계산 로직을 SQL이 아닌 자바 코드 수준에서 표현할수있도록 구현했습니다. 
쿼리를 직접 작성 하는것보다 API 로직의 의도가 코드 수준에서 표현하는것이 의도가 명확히 보이는 장점이 있기도 하고 테스트 작성도 수월해집니다.

### 🔐 Redis 인증 처리

회원가입 이메일 인증 번호를 Redis 저장소에 저장하도록 구현했습니다. 
해당 이유는 TTL에 있습니다. TTL을 설정해두면 유효 시간이 지난 인증 번호는 별도의 삭제 작업 없이 자동으로 사라집니다. 
RDBMS로 관리할 경우 만료된 레코드를 주기적으로 정리하는 batch 작업이 필요한데, Redis는 그 과정이 필요 없기 때문에 매우 편리하고 용이합니다.

응답 속도 측면에서도 인메모리 저장소는 디스크 기반 데이터베이스와 비교가 안 됩니다. 인증 번호 검증에서 체감할 수 있는 지연이 거의 없고, 회원가입 흐름에서 이탈하는 사용자를 줄이는 데도 영향을 줍니다.

### 🔐 JWT 및 리프레시 토큰

액세스 토큰 만료 시간은 30분입니다. 탈취되더라도 피해가 제한됩니다. 리프레시 토큰은 14일 유효합니다. 30분마다 재로그인하지 않아도 됩니다. 두 토큰은 HttpOnly 쿠키에 담겨 전달되기 때문에 자바스크립트로 접근할 수 없습니다. XSS 공격으로 토큰을 탈취하는 경로를 막는 방식입니다.

보안을 높이면 사용자 경험이 나빠지는 경우가 많습니다. 만료 시간을 짧게 잡으면 자주 재로그인해야 하고, 길게 잡으면 탈취 위험이 커집니다. 액세스 토큰과 리프레시 토큰을 함께 운용하는 이유가 여기 있습니다.

## 1.3 Goal

분석 단계의 문서는 기획을 구현으로 연결하는 번역 작업입니다. 아이디어가 아무리 좋아도 구체화되지 않으면 코드로 옮길 수 없고, 이 단계에서 넘긴 빈틈은 개발 중에 반드시 더 큰 비용으로 돌아옵니다. 본 문서는 네 가지 목표를 기준으로 작성됩니다.

### 1.3-1 요구사항의 기술적 구체화

유스케이스 다이어그램과 상세 기술서로 기능을 정형화합니다. 경험치 계산 하나만 봐도 개념 수준의 설명으로는 구현이 불가능합니다. 종목별 기본 경험치가 얼마인지, 세트 수와 반복 횟수에 어떤 가중치를 적용하는지, 하루 중복 기록이 들어오면 어떻게 처리하는지, 보너스 조건이 충족되는 시점이 언제인지까지 규칙으로 정의되어야 합니다. 이 과정에서 미처 생각하지 못한 엣지 케이스가 드러나고, 개발 전에 해결함으로써 나중의 재작업을 줄입니다.

### 1.3-2 데이터 무결성 및 흐름 정의

운동 데이터가 입력되고 저장하고 경험치로 바뀌고, 다시 화면에 표시되는 과정 전체를 따라가 봅니다. 
사용자 입장에서 운동 기록은 단순한 숫자가 아닙니다. 실제로 땀을 흘리며 자신을 업그레이드 해나가는 과정입니다
그래서 어느 한 단계라도 데이터가 이상해진다면, 사용자는 나의 노력은 도대체 어디로 갔는것인가 하며 회의감이 들며 운동 하는 횟수가 줄어 들게 됩니다.

그렇기 때문에 경험치 계산과 티어 갱신은 운동 기록 저장과 반드시 하나의 트랜잭션으로 묶여야 합니다. 기록은 저장됐는데 경험치 반영이 안 된 채로 끝나는 상황 즉 부분적인 작업성공이 일어나서는 안됩니다.

### 1. 액터 계층 구조: 일반화(Generalization)

다이어그램 왼쪽을 보면 사용자 계층이 세 단계로 구성되어 있습니다. 최상위에 위치한 User는
Ex-Log 서비스에 접근하는 모든 사람을 아우르는 루트 액터입니다. 회원가입, 로그인, 마이페이지
조회처럼 서비스 전반에 걸친 범용 기능들이 이 레벨에 연결됩니다.

그 아래 Strength Trainee와 Cardio Trainee는 User를 상속합니다. 공통된 권한은 물려받되,
근력 운동과 유산소 운동이라는 각자의 영역에서만 유효한 행위를 독립적으로 수행합니다.
마치 공통 부품을 공유하는 두 개의 제품 라인처럼, 기반을 공유하면서 각자의 방향으로
뻗어나가는 구조입니다. 사용자 유형이 훗날 늘어나더라도 공통 기능을 처음부터 재정의할
필요 없이 자식 액터만 확장하면 됩니다.

외부 액터인 Email Server는 시스템 경계 너머에 배치해, 인증을 보조하는 역할에만 집중하도록
그 범위를 명확히 한정했습니다.

### 2. 보안 설계: 인증의 선행 강제

오른쪽 흐름에서 Login to System으로 뻗어나가는 include 점선들이 눈에 띕니다.
회원가입을 제외하면 기록 입력, 조회, 도감 열람 등 사실상 모든 핵심 기능이
로그인을 전제로 작동합니다. 이는 단순한 UX 설계가 아닙니다. 백엔드 구현 단계에서
필터(Filter) 기반의 인증·인가 처리가 반드시 동반되어야 한다는 점을, 설계 초기에
이미 못 박아둔 결정입니다.

Register Account가 Verify Identity via Email을 포함(include)하는 구조도
같은 맥락입니다. 이메일 인증을 회원가입의 필수 절차로 내재화함으로써,
봇이나 비정상적인 경로를 통한 계정 생성 시도를 흐름 자체에서 차단합니다.

### 3. 운동 기록 흐름: 단계의 강제적 순서

운동 데이터를 입력하는 경로는 실제 사용자의 행동 순서를 그대로 모델링했습니다.
Input Data → Select Specific Exercise → Select Exercise Category로 이어지는
include 체인은, 카테고리와 종목을 먼저 선택해야만 수치 입력 화면에 진입할 수 있도록
강제합니다.

이 구조가 없다면 종목 정보 없이 숫자만 덩그러니 저장되는 상황이 발생할 수 있습니다.
DB 저장 시점의 정합성 오류를 코드가 아닌 설계 단계에서 미리 봉쇄한 셈입니다.
뼈대를 단단히 세워두면, 나중에 살을 붙이는 작업이 훨씬 안전해집니다.

### 4. 부가 기능 분기: 확장(Extend)의 선택적 흐름

마이페이지 영역에서는 include 대신 extend를 채택했습니다. 대시보드를 조회하는 것은
필수 행위지만, 보유 칭호를 확인하거나 대표 칭호를 지정하는 것은 어디까지나
사용자의 자발적 선택입니다. extend는 바로 이 '선택의 여지'를 표현하는 데 적합한 관계입니다.

MyPage → View Acquired Titles → Set Representative Title로 이어지는 진입 경로는
명확하게 유지하면서도, 각 단계를 강요하지 않습니다. 구조적 일관성을 유지하되
사용자에게 자유도를 돌려주는, 균형 잡힌 설계 지점입니다.

| Title | 카테고리별 획득 조건과 임계값을 정의하는 칭호 메타 정보 엔티티 |
| UserTitle | User와 Title 간의 칭호 획득 관계 및 획득 시점을 매핑하는 중간 엔티티 |
| Tier | 누적 경험치에 따른 사용자 등급을 표현하는 열거형(Enum) |
| TitleCondition | 칭호 획득 조건의 유형을 분류하는 열거형(Enum) |

## 5. Glossary

| **Term** | **Description** |
| --- | --- |
| **점진적 과부하** | 근육 성장을 위해 운동 강도나 볼륨을 체계적으로 늘려가는 원리. 본 서비스의 핵심 지향점 |
| **EXP (Experience)** | 사용자의 활동량에 따라 부여되는 경험치 수치. 티어 상승의 척도 |
| **Tier (티어)** | 누적 경험치에 따라 분류되는 사용자의 등급 (예: Bronze, Silver, Gold 등) |
| **Streak (잔디)** | 연속적인 운동 참여를 시각화한 활동 지도. 동기부여를 위한 핵심 UI 요소 |
| **칭호 (Title)** | 특정 운동 미션 달성 시 부여되는 디지털 업적 보상 |
| **유즈케이스 (Use Case)** | 사용자 관점에서 시스템이 제공하는 기능적 요구사항을 시나리오 형태로 기술한 것 |
| **인증 (Authentication)** | 이메일 및 비밀번호를 통해 정당한 사용자임을 확인하는 보안 절차 |
