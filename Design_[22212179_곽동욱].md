# 1. Introduction
이 문서는 현대인의 운동 동기부여를 위한 백엔드 시스템 'Ex-Log'의 상세 설계 내용을 담고 있습니다. 사용자 경험(UX)을 극대화하기 위한 티어 시스템, 칭호 도감, 그리고 보안과 성능을 고려한 인증 로직이 이 설계의 핵심입니다.

## 📑 1. 문서 요약 (Document Summary)
본 설계 문서는 Spring Boot와 Java를 기반으로 한 운동 기록 플랫폼의 전반적인 아키텍처를 정의합니다. 주요 내용은 다음과 같습니다.

시스템 구조: 계층형 아키텍처(Layered Architecture)를 채택하여 유지보수성과 확장성을 확보했습니다. 특히 Swagger API 문서와 Controller를 분리하여 코드의 가독성을 높였습니다.

핵심 도메인: 회원 인증(Auth), 운동 기록(Exercise Log), 보상 시스템(EXP & Tier), 칭호(Title), 마이페이지(MyPage)로 구성되어 있습니다.

데이터 흐름: 클래스 다이어그램 및 시퀀스 다이어그램을 통해 객체 간의 협력 관계와 시간 순서에 따른 메시지 흐름을 상세히 기술했습니다.

보안 및 인프라: JWT를 이용한 무상태(Stateless) 인증 방식과 Redis를 활용한 이메일 검증 및 토큰 블랙리스트 관리 전략을 포함합니다.

## 💡 2. 설계의 핵심 포인트 (Key Design Points)
개발자로서 시스템의 안정성과 협업 효율을 고려해 다음의 4가지 포인트에 집중하여 설계했습니다.

### ① 계층 간 책임 분리와 가독성 (Layered Architecture & Javadoc)
백엔드 개발자로서 코드의 깔끔함과 협업 효율을 최우선으로 했습니다. 모든 서비스 로직은 도메인별로 철저히 분리하였으며, Swagger 어노테이션이 Controller의 비즈니스 로직 시인성을 해치지 않도록 인터페이스나 별도 클래스로 분리하여 설계했습니다. 또한, 모든 Java 메소드에는 Javadoc을 작성하여 동료 개발자가 로직의 의도와 파라미터를 즉각적으로 이해할 수 있도록 구성했습니다.

### ② Redis를 통한 성능 및 보안 최적화 (Redis Integration)
데이터베이스(RDB)의 부하를 줄이고 보안을 강화하기 위해 Redis를 적극적으로 활용했습니다.

이메일 인증: 인증번호와 가입 완료 티켓을 Redis의 TTL(Time-To-Live) 기능을 이용해 관리함으로써, 불필요한 데이터가 DB에 남지 않도록 설계했습니다.

로그아웃 처리: 로그아웃된 Access Token을 Redis 블랙리스트에 등록하여, 토큰 탈취 시에도 즉각적인 무효화가 가능하도록 보안 수준을 높였습니다.

### ③ 확장 가능한 보상 및 티어 알고리즘 (Reward & Tier Logic)
단순한 기록 저장을 넘어 사용자의 동기부여를 위한 보상 로직을 정교하게 설계했습니다.

보상은 동기부여를 주기에는 확실하고 최고라고 생각하여 보상으로 티어 및 경험치를 설계 했습니다.

서비스 간 협력: ExerciseLogService가 기록을 저장한 뒤, ExpService와 UserTitleService를 순차적으로 호출하는 구조를 통해 기록-보상-칭호 획득으로 이어지는 연쇄 로직을 구현했습니다.

조건별 보상: '하루 3종목 이상 완료 시 추가 경험치(+5)'와 같은 세부 비즈니스 요구사항을 처리하기 위해 서비스 간의 결합도를 낮추고 각자의 책임에 집중하게 했습니다.

### ④ 데이터 정합성과 예외 처리 (Data Integrity & Exception)
데이터의 무결성을 보장하기 위해 모든 상태 변경 작업은 @Transactional 범위 내에서 관리됩니다.

검증 로직: 회원가입 시 이메일 인증 여부 확인, 운동 기록 시 카테고리별(유산소/무산소) 필수 입력값(시간, 세트 등) 검증을 서비스 레이어 진입 시점에서 철저히 수행하도록 설계했습니다.

사용자 정의 예외: CustomException과 ErrorCode 체계를 구축하여, 클라이언트에게 일관된 에러 응답을 제공하고 디버깅 효율을 높였습니다.

# 3. Sequence Diagram

## 1) SignUp
![signup](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/7.Signup.png)

### Description
회원가입 기능은 새로운 사용자를 Ex-Log 시스템에 정식으로 등록하기 위한 일련의 검증 및 영속화 프로세스입니다. 단순한 사용자 정보 저장에 그치지 않고, 이메일 실존 여부 확인부터 비밀번호 암호화, 임시 인증 데이터 정리에 이르기까지 다단계 보안 절차를 포함하는 체계적인 인증 진입 지점으로 설계되었습니다.
사용자는 이메일 주소, 비밀번호, 닉네임 등의 필수 정보를 입력하여 회원가입 요청을 전송합니다. 

해당 요청이 서버에 도달하면 시스템은 가장 먼저 이메일 중복 여부를 검사합니다. 이미 동일한 이메일로 가입된 계정이 존재하는 경우, 시스템은 즉시 가입 불가 응답을 반환하여 중복 계정 생성을 원천 차단합니다.
이메일 중복이 확인되지 않은 경우, 시스템은 다음 단계로 해당 이메일에 대한 인증 완료 여부를 검증합니다. 이는 이메일 인증 단계(1-1, 1-2)에서 생성된 인증 상태 데이터를 조회하여 판별하며, 인증이 정상적으로 완료된 이메일에 한해서만 회원가입 절차가 진행됩니다. 인증을 거치지 않은 이메일로 가입을 시도하는 경우 요청이 거부되며, 이를 통해 허위 이메일 주소를 통한 가입을 방지합니다.


두 가지 검증을 모두 통과한 경우, 사용자가 입력한 평문 비밀번호는 단방향 해시 암호화 알고리즘을 통해 안전한 형태로 변환됩니다. 암호화된 비밀번호와 나머지 사용자 정보는 함께 데이터베이스에 영속화되며, 어떠한 경우에도 평문 비밀번호 원본은 시스템 내부 어디에도 저장되지 않습니다.
회원 정보 저장이 완료된 직후, 시스템은 인증 과정에서 임시로 생성 및 보관되었던 인증번호 및 인증 상태 데이터를 즉시 삭제합니다. 

이 정리 단계는 동일한 인증 토큰이나 인증 상태를 재사용하여 비정상적인 가입 요청을 수행하는 공격 시나리오를 방지하기 위한 목적으로 설계된 보안 처리입니다.
모든 과정이 성공적으로 완료되면 시스템은 회원가입 완료 응답을 반환하며, 신규 사용자는 이후 로그인 기능을 통해 Access Token을 발급받아 Ex-Log의 운동 기록, 칭호 수집, 마이페이지 등 인증 기반 기능 전반을 이용할 수 있게 됩니다. 이처럼 회원가입 기능은 단순 등록 과정이 아닌, 이메일 인증과 보안 검증을 통합한 신뢰 기반 사용자 관리 체계의 시작점으로 기능합니다.

## 1-1) Verify Code Send
![verifycodesend](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/7-1.VerifyCodeSend.png)

### Description
이메일 인증번호 발급 기능은 회원가입 요청 이전에 사용자가 제출한 이메일 주소의 실존 여부 및 실제 소유 여부를 확인하기 위한 사전 인증 메커니즘입니다. 

시스템 입장에서는 임의로 조작되거나 타인의 이메일을 도용한 가입 시도를 사전에 차단할 수 있는 핵심적인 보안 장치로 작동합니다.
사용자가 이메일 주소를 입력하고 인증번호 발송을 요청하면, 시스템은 서버 측에서 난수 기반의 인증번호를 생성합니다. 생성된 인증번호는 유효 시간이 설정된 상태로 서버 측 임시 저장소(예: Redis 등의 인메모리 저장소)에 이메일 주소와 함께 매핑되어 저장됩니다. 이 유효 시간 제한은 인증번호가 외부로 유출되더라도 일정 시간이 경과하면 자동으로 사용 불가 상태가 되도록 함으로써 인증번호의 남용 가능성을 최소화합니다.


저장이 완료된 직후, 시스템은 메일 전송 모듈을 통해 해당 이메일 주소로 인증번호를 발송합니다. 발송되는 이메일에는 인증번호와 함께 유효 시간 안내 메시지가 포함되어 사용자가 시간 내에 인증을 완료할 수 있도록 유도합니다.
이 기능은 실제 이메일 계정에 접근 가능한 사용자만이 이후 인증번호 검증 단계를 통과하여 회원가입으로 진행할 수 있도록 하는 구조적 보장 장치입니다. 또한 인증번호가 서버에서 직접 생성되어 이메일로만 전달되는 방식이므로, 클라이언트 측에서 인증 결과를 임의로 조작할 수 없는 서버 주도형 인증 흐름을 구현합니다.

## 1-2) Verify Code
![verify](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/7-2.Verify.png)

### Description
이메일 인증번호 검증 기능은 사용자가 수신한 인증번호를 실제로 입력하여 인증 절차를 완결짓는 단계로, 이메일 소유권을 최종 확인하는 과정입니다. 이 단계가 성공적으로 완료되어야만 이후 회원가입 요청이 유효한 것으로 처리됩니다.


사용자가 이메일 주소와 인증번호를 함께 제출하면, 시스템은 해당 이메일 키를 기반으로 임시 저장소에 보관된 인증번호를 조회합니다.

조회된 인증번호와 사용자 입력값을 비교하여 두 값이 일치하는지 검사하며, 동시에 인증번호의 유효 시간이 만료되지 않았는지도 함께 확인합니다. 인증번호가 만료된 경우에는 값이 일치하더라도 인증 실패로 처리되어, 시간 기반 보안 정책이 엄격히 적용됩니다.


검증이 성공하면 시스템은 해당 이메일에 대한 인증 완료 상태를 저장합니다. 이 상태 정보는 이후 회원가입 단계에서 이메일 인증 여부를 판별하는 근거로 활용됩니다. 동시에 검증에 사용된 인증번호 원본 데이터는 즉시 삭제되어, 동일 인증번호를 이용한 반복적인 인증 시도나 재사용 공격을 원천 차단합니다.
  
  
반대로 인증번호가 불일치하거나 이미 만료된 경우에는 인증 실패 응답이 반환되며, 사용자는 인증번호를 재발급받아 과정을 다시 진행해야 합니다. 이처럼 인증번호의 단회성 처리와 시간 제한 정책은 이 기능의 보안성을 뒷받침하는 핵심 설계 원칙입니다.

## 2) Login
![login](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/1.Login.png)

### Description
로그인 기능은 이미 등록된 사용자가 자신의 자격증명(Credential)을 통해 Ex-Log 시스템에 대한 접근 권한을 획득하는 인증 과정입니다. 이 과정은 단순한 사용자 확인에 그치지 않고, 이후 모든 보호된 API 요청의 기반이 되는 토큰 쌍을 발급하는 세션 초기화 절차로서의 역할을 수행합니다.


사용자는 이메일과 비밀번호를 입력하여 로그인 요청을 전송합니다. 요청은 컨트롤러 계층을 통해 서비스 계층으로 전달되며, 서비스에서는 입력된 이메일에 대응하는 계정이 데이터베이스에 존재하는지 먼저 확인합니다. 해당 계정이 존재하지 않는 경우, 시스템은 사용자 미존재 응답을 반환합니다.
계정이 확인되면 입력된 비밀번호를 저장된 암호화 비밀번호와 비교하는 검증 단계가 수행됩니다. 비밀번호는 단방향 해시 방식으로 저장되어 있으므로, 입력값 역시 동일한 알고리즘으로 변환된 후 저장값과 비교됩니다. 불일치 시에는 인증 실패 응답이 반환됩니다.


인증이 성공하면 시스템은 두 종류의 JWT 토큰을 생성합니다. Access Token은 짧은 유효 시간을 가지며, 이후 인증이 요구되는 모든 API 요청 시 Authorization 헤더에 포함되어 전달됩니다. Refresh Token은 상대적으로 긴 유효 시간을 가지며, Access Token이 만료되었을 때 재발급을 요청하는 데 사용됩니다. 
  
 Refresh Token은 서버 측 저장소에 안전하게 보관되어 이중 검증 구조를 형성하며, 클라이언트가 보유한 Refresh Token과 서버에 저장된 값을 대조함으로써 토큰 위조 및 탈취 시도에 대응합니다.
최종적으로 시스템은 로그인 성공 응답과 함께 발급된 토큰 정보를 클라이언트에 반환하며, 사용자는 이후 운동 기록, 칭호 조회, 마이페이지 접근 등 인증 기반의 모든 서비스 기능을 정상적으로 이용할 수 있게 됩니다.

## 3) Logout
![logout](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/2.Logout.png)

### Description
로그아웃 기능은 현재 인증 상태에 있는 사용자의 세션을 안전하게 종료하고, 발급된 토큰의 사용 가능성을 서버 측에서 완전히 차단하는 보안 처리 절차입니다. 단순히 클라이언트에서 토큰을 삭제하는 방식에 의존하지 않고, 서버 레벨에서 능동적으로 인증 무효화를 수행하는 구조로 설계되었습니다.
사용자가 로그아웃을 요청하면 현재 사용 중인 Access Token과 함께 요청이 서버로 전달됩니다. 서버는 먼저 해당 사용자와 연결된 Refresh Token을 저장소에서 삭제합니다. 이 처리를 통해 로그아웃 이후에는 만료된 Access Token을 이용한 재발급 요청이 원천적으로 불가능해집니다.


JWT는 토큰 자체에 인증 정보와 유효 시간이 포함된 무상태(Stateless) 방식으로 작동하기 때문에, 서버가 토큰의 유효성을 개별적으로 추적하지 않는 기본 구조에서는 유효 시간이 남은 토큰이 탈취되었을 경우 이를 즉시 무력화할 방법이 없습니다. 

이 문제를 해결하기 위해 시스템은 로그아웃된 Access Token을 별도의 블랙리스트 저장소에 등록합니다. 이후 해당 토큰으로 들어오는 모든 API 요청은 인증 필터 계층에서 블랙리스트 여부를 먼저 확인하여 요청을 거부하는 방식으로 처리됩니다.
서버 측 처리가 완료되면 클라이언트에게 로그아웃 완료 응답이 반환되며, 클라이언트는 로컬에 저장된 Access Token 및 Refresh Token 정보를 함께 제거합니다. 
  
이후 사용자는 인증이 필요한 모든 기능에 접근할 수 없게 되며, 시스템은 사용자를 로그인 화면으로 리다이렉트하여 인증 종료 상태를 명확히 안내합니다. 이와 같이 서버와 클라이언트 양측을 동시에 처리하는 이중 종료 구조는 JWT 기반 인증 시스템에서 완전한 로그아웃을 보장하기 위한 필수적인 설계 방식입니다.

## 4) Exercise Logging
![logging](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/3.Logging.png)

### Description
운동 기록 기능은 Ex-Log 시스템의 핵심 도메인 기능으로, 사용자의 운동 활동 데이터를 저장하는 것을 넘어 경험치 계산, 티어 승급, 잔디 시스템 반영, 칭호 획득 판정 등 다양한 부가 기능을 연쇄적으로 트리거하는 중추적인 역할을 수행합니다.
사용자는 먼저 운동 카테고리를 선택합니다. 카테고리는 가슴, 등, 하체, 어깨, 팔, 유산소 등 신체 부위 및 운동 유형에 따라 구분되며, 카테고리 선택 시 해당 카테고리에 속하는 운동 종목 목록이 동적으로 로드됩니다. 
사용자는 목록에서 실제로 수행한 운동 종목을 선택한 뒤, 운동 유형에 맞는 데이터를 입력합니다.


입력 데이터는 운동 유형에 따라 다음과 같이 구분됩니다:
1. 무산소 운동 (저항 운동): 세트 수(Set), 세트당 반복 횟수(Rep)
2. 유산소 운동 (Cardio): 운동 지속 시간, 이동 거리


입력된 데이터는 서비스 계층에서 유효성 검사를 거친 뒤 데이터베이스에 영속화됩니다. 저장 완료 시점을 기준으로 여러 연계 처리가 순차적으로 수행됩니다.


**잔디 시스템(Contribution Graph)**: 운동 기록이 저장되면 해당 날짜의 잔디가 활성화됩니다. 잔디는 사용자의 운동 활동 여부를 날짜 단위로 시각화하는 요소로, 연속 기록 유지를 유도하는 동기 부여 수단으로 기능합니다.
**경험치 시스템**: 운동 기록 저장 시 기본 경험치가 지급됩니다. 추가적으로, 하루 동안 서로 다른 운동 종목을 3개 이상 수행한 것으로 판정될 경우 보너스 경험치가 추가 지급됩니다. 이 정책은 다양한 신체 부위를 균형 있게 단련하도록 유도하기 위한 설계 의도를 담고 있습니다.
**티어 시스템**: 누적된 경험치는 사용자의 현재 티어와 연계되어, 일정 수준 이상 달성 시 자동으로 상위 티어로 승급 처리됩니다. 티어는 사용자의 전반적인 운동 활동량과 꾸준함을 반영하는 등급 지표입니다.
**칭호 시스템 연동**: 운동 기록은 칭호 획득 조건 판정 로직과도 연결됩니다. 특정 운동 종목의 누적 횟수 달성, 연속 운동 일수 유지 등 사전에 정의된 조건을 만족한 경우, 시스템은 해당 칭호를 사용자에게 자동으로 부여합니다.
  
  
이처럼 운동 기록 하나의 처리 흐름이 다수의 도메인 로직을 연쇄적으로 실행하는 구조이므로, 트랜잭션 일관성 유지와 각 처리 단계 간의 순서 보장이 중요한 설계 고려 사항입니다.

## 5) All Title Encyclopedia
![alltitle](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/4.AllTitle.png)

### Description
전체 칭호도감 조회 기능은 시스템에 정의된 모든 칭호 정보를 사용자에게 제공하는 기능으로, 사용자가 현재 자신의 수집 현황을 파악하고 이후 달성 목표를 구체적으로 설정할 수 있도록 지원합니다. 

단순 목록 조회를 넘어, 획득 여부를 구분하여 표시하는 방식으로 사용자 경험과 게이미피케이션(Gamification) 요소를 결합한 기능입니다.
시스템은 요청 수신 시 두 가지 데이터를 병행 조회합니다. 하나는 시스템에 등록된 전체 칭호 목록이며, 다른 하나는 해당 사용자가 현재까지 획득한 칭호 목록입니다. 

두 데이터를 비교하여 각 칭호에 대한 획득 여부를 플래그로 표시한 뒤, 이를 통합된 응답 구조로 반환합니다.
응답에서 획득 완료된 칭호는 활성화 상태로 표시되며, 아직 획득하지 못한 칭호는 잠금 상태로 구분 표시됩니다. 

잠금 상태의 칭호에는 획득 조건 정보가 함께 제공됩니다. 예를 들어 '특정 운동 종목 100회 달성', '30일 연속 운동 성공', '5가지 서로 다른 운동 카테고리 기록' 등의 조건을 명시하여, 사용자가 어떠한 활동을 통해 해당 칭호를 획득할 수 있는지 명확하게 안내합니다.


이 기능은 단순한 수집 콘텐츠를 제공하는 데 그치지 않고, 사용자가 보유하지 못한 칭호의 조건을 인지함으로써 다음 운동 세션의 방향성과 동기를 부여하는 구조적 역할을 수행합니다. 전체 칭호도감은 Ex-Log의 게이미피케이션 설계에서 사용자의 장기적 참여를 유도하는 핵심 콘텐츠 허브로 기능합니다.

---

## 6) Log History
![loghistory](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/5.LogHistory.png)

### Description
날짜별 상세 기록 조회 기능은 사용자가 과거에 수행한 운동 데이터를 시간 순서에 따라 열람할 수 있도록 제공되는 이력 관리 기능입니다. 단순한 기록 열람을 넘어, 운동 패턴의 장기적 분석과 자기 관리의 근거 데이터로 활용될 수 있도록 설계되었습니다.


사용자는 마이페이지 내 운동 기록 메뉴를 통해 자신의 기록 이력에 접근합니다. 조회 요청이 서버로 전달되면 시스템은 해당 사용자에 귀속된 운동 기록을 최신 순으로 정렬하여 반환합니다. 
대량의 기록 데이터를 한 번에 반환할 경우 응답 크기와 처리 부하가 증가하는 문제를 방지하기 위해, 페이징(Paging) 방식을 적용하여 일정 단위로 분할하여 제공합니다. 
사용자는 스크롤 또는 페이지 이동을 통해 이전 기록으로 순차적으로 접근할 수 있습니다.


각 날짜별 기록에는 다음 정보가 포함됩니다:
**운동 수행 날짜**: 기록이 저장된 날짜
**운동 종목**: 수행된 운동의 명칭
**세트 수 및 반복 횟수**: 무산소 운동에 해당
**운동 시간 및 이동 거리**: 유산소 운동에 해당


이 기능을 통해 사용자는 특정 기간 동안 자신이 얼마나 꾸준히 운동했는지, 어떤 운동 종목을 주로 수행했는지, 특정 종목의 수행 빈도나 운동량의 변화 추이는 어떠한지를 직접 확인할 수 있습니다. 이는 단순한 과거 기록 열람을 넘어, 사용자 스스로 운동 루틴을 분석하고 개선 방향을 수립할 수 있는 데이터 기반 셀프 코칭의 토대로 기능합니다.

## 7) My Title & Main Title Setup
![mytitlemaintitle](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/6.MyTitleMainTitle.png)

### Description
대표 칭호 설정 기능은 사용자가 자신이 보유한 칭호 목록 중 하나를 대표 칭호(Main Title)로 지정하여 프로필에 표시할 수 있도록 제공되는 사용자 커스터마이징 기능입니다. 

칭호 획득이 단순한 보상 수령에서 끝나는 것이 아니라, 자신의 운동 성과를 공개적으로 표현하는 수단으로 확장되는 구조입니다.
사용자가 대표 칭호 설정 화면에 진입하면, 시스템은 해당 사용자가 현재까지 획득한 칭호 목록을 조회하여 반환합니다. 사용자는 목록에서 원하는 칭호를 선택하여 대표 칭호 변경 요청을 전송합니다.


요청이 서버에 도달하면 시스템은 두 가지 사항을 검증합니다. 첫째, 사용자가 요청한 칭호가 실제로 해당 사용자의 획득 목록에 존재하는지 확인합니다. 타 사용자의 칭호 ID를 임의로 삽입하거나, 미획득 칭호를 직접 지정하는 방식의 변조 요청을 방지하기 위한 서버 측 소유권 검증입니다.

둘째, 현재 이미 동일한 칭호가 대표 칭호로 설정되어 있는 중복 요청이 아닌지 판별하여 불필요한 DB 갱신 연산을 최소화합니다.
검증이 완료되면 해당 사용자의 대표 칭호 정보가 데이터베이스에 갱신됩니다. 변경된 대표 칭호는 사용자 프로필 및 마이페이지에 즉시 반영되어, 다른 사용자에게 해당 사용자의 대표적인 성취로 표시됩니다.


이 기능은 운동 성취에 사회적 표현 요소를 부가함으로써 사용자의 성취감을 강화하는 동시에, 향후 사용자 간 프로필 비교나 커뮤니티 기능이 확장될 경우 경쟁 및 소통의 핵심 정체성 요소로 발전할 수 있도록 설계되었습니다.

## 8) Streak Mail
![mytitlemaintitle](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/Streakmail.png)

### Description 
미접속 유저 스트릭 경고 메일 발송 기능은 사용자의 운동 스트릭(Streak) 유지율을 높이기 위해, 일정 기간 동안 운동 기록이 없는 사용자에게 자동으로 경고 메일을 발송하는 배치 시스템 기능입니다.

운동 습관 형성 서비스에서 스트릭은 사용자의 지속적인 참여를 유도하는 핵심 요소이며, 일정 기간 활동이 중단될 경우 스트릭이 끊길 가능성이 높아집니다.
본 기능은 이러한 이탈 가능 사용자를 조기에 탐지하여 운동 복귀를 유도하기 위한 리텐션(Retention) 목적의 자동 알림 시스템으로 설계되었습니다.

시스템 스케줄러는 매일 오전 9시에 자동 실행되며, 데이터베이스에서 마지막 운동 기록이 3일 전인 사용자를 조회합니다. 조회 조건은 서비스 정책에 따라 조정 가능하며, 현재는 스트릭 유지 임계 구간에 진입한 사용자를 기준으로 설정되어 있습니다.

조회된 사용자 목록은 반복 순회를 통해 개별적으로 처리되며, 각 사용자에게 운동 지속을 유도하는 스트릭 경고 메일 발송 요청이 전달됩니다. 메일 발송 과정은 @Async 기반 비동기 처리로 구성되어 있어, 다수 사용자에게 메일을 전송하더라도 전체 스케줄러 작업이 지연되지 않도록 설계되었습니다.

메일 서비스에서는 사용자 정보를 기반으로 메일 제목과 템플릿 내용을 생성합니다. 메일에는 현재 스트릭 상태, 운동 재참여 유도 문구, 서비스 복귀를 위한 안내 메시지 등이 포함될 수 있으며, 최종적으로 JavaMailSender를 통해 SMTP 서버로 실제 메일이 전송됩니다.

메일 전송 과정에서는 SMTP 연결 실패, 잘못된 이메일 주소, 네트워크 오류 등 다양한 예외 상황이 발생할 수 있으므로, 시스템은 전송 실패에 대한 예외 처리 및 로그 기록 기능을 함께 수행합니다. 이를 통해 운영자는 메일 발송 실패 여부를 추적하고 안정적인 배치 운영 상태를 유지할 수 있습니다.

이 기능은 단순 알림 기능을 넘어, 사용자의 운동 습관 유지와 서비스 재방문율 향상을 위한 핵심 리텐션 전략으로 활용됩니다. 또한 향후에는 사용자 활동 패턴 분석 기반 개인화 알림, 운동 추천 메시지, 푸시 알림 연동 등의 기능으로 확장될 수 있도록 고려하여 설계되었습니다.

# 4. State Machine Diagram

# 1) User State Diagram

![userstate](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/Userstate.png)

## Description  
User 기반 전역 시스템 상태 정의서는 Ex-Log 서비스에서 사용자가 시스템에 진입한 이후 로그아웃할 때까지의 전체 서비스 흐름과 상태 전이를 통합적으로 표현한 상태도(State Machine Diagram)입니다.

본 상태도는 인증, 운동 기록, 보상 처리, 마이페이지, 칭호 관리 등 서비스 전반의 핵심 기능들을 사용자 중심 흐름(User-Centric Flow) 기준으로 구조화하여 표현하고 있으며, 각 기능이 독립적으로 동작하면서도 하나의 통합된 사용자 경험으로 연결되도록 설계된 시스템 구조를 나타냅니다.

전체 시스템은 크게 다음과 같은 다섯 개의 영역으로 구성됩니다.

- 인증 및 회원가입(Auth Section)
- 메인 대시보드(Main)
- 운동 기록 시스템(Exercise Section)
- 마이페이지 및 칭호 관리(Management Section)
- 시스템 종료(Logout)

각 영역은 사용자의 행동에 따라 상태가 전이되며, 특정 작업 수행 이후 다시 메인 상태로 복귀하는 허브(Hub) 구조 기반으로 설계되어 있습니다.

---

### 1. 인증 및 회원가입 섹션 (Auth Section)

시스템의 시작 상태는 인증(Auth) 영역입니다.

초기 상태에서는 사용자가 `Login` 상태로 진입하게 되며, 해당 상태에서는 아이디(ID)와 비밀번호(PW) 입력을 대기합니다.

```text
state Login : do/ID, PW 입력 대기
```

사용자는 로그인에 성공할 경우 메인 대시보드 상태로 이동하게 되며, 아직 회원이 아닌 경우 회원가입(Join) 상태로 진입할 수 있습니다.

회원가입 프로세스는 이메일 인증 기반 구조로 설계되어 있습니다.

회원가입 흐름은 다음과 같은 세부 상태로 구성됩니다.

### 1-1. EmailInput

사용자가 이메일 주소를 입력하는 단계입니다.

시스템은 입력된 이메일 형식 검증 및 중복 여부를 검사할 수 있으며, 유효한 경우 인증 코드 발송 단계로 이동합니다.

---

#### 1-2. CodeWaiting

사용자가 이메일 인증번호를 입력하는 상태입니다.

시스템은 메일 서버를 통해 발송된 인증 코드와 사용자가 입력한 값을 비교하여 인증 여부를 검증합니다.

---

#### 1-3. EmailVerified

이메일 인증이 완료된 상태입니다.

이 단계에서는 계정 생성 승인 대기 상태로 전환되며, 이후 회원가입 완료 처리 후 로그인 상태로 복귀합니다.

이메일 인증 기반 회원가입 구조는 다음 목적을 가집니다.

- 중복 계정 생성 방지
- 허위 이메일 가입 제한
- 비밀번호 찾기 및 계정 복구 기반 확보
- 알림 메일 시스템 연동 기반 구축

회원가입이 완료되면 사용자는 다시 로그인 상태로 이동하여 인증 절차를 진행하게 됩니다.

---

### 2. 메인 대시보드 (Main)

메인 대시보드는 전체 시스템의 중앙 허브(Hub) 역할을 수행하는 핵심 상태입니다.

```text
state Main : entry/유저 정보 동기화
do/티어, 경험치, 잔디 렌더링
```

로그인 성공 시 사용자는 Main 상태로 진입합니다.

Main 상태에서는 다음과 같은 주요 작업이 수행됩니다.

- 사용자 정보 동기화
- 경험치 및 티어 조회
- 운동 잔디 데이터 렌더링
- 대표 칭호 및 프로필 정보 출력
- 최근 운동 활동 표시

즉, 사용자 개인화 데이터가 집약적으로 표시되는 메인 화면입니다.

또한 Main 상태는 다음 기능 영역으로 이동하는 중심 노드 역할을 수행합니다.

- 운동 기록 시스템
- 마이페이지
- 칭호 관리
- 로그아웃

이러한 구조는 사용자 흐름 단순화와 UI 접근성 향상을 목적으로 설계되었습니다.

---

### 3. 운동 기록 섹션 (Exercise Section)

운동 기록 섹션은 사용자의 실제 운동 데이터를 저장하고 보상 시스템과 연결되는 핵심 기능 영역입니다.

사용자가 메인 화면에서 “운동 시작” 버튼을 클릭하면 운동 기록 상태로 이동합니다.

```text
Main -right-> CategorySelect : click 운동 시작
```

---

#### 3-1. CategorySelect

운동 카테고리 선택 단계입니다.

사용자는 운동 부위를 기준으로 카테고리를 선택하게 됩니다.

예시:
- 가슴
- 등
- 하체
- 어깨
- 유산소

해당 단계에서는 운동 목록 필터링 기능이 수행됩니다.

---

#### 3-2. ExerciseSelect

카테고리 내부의 운동 종목을 선택하는 단계입니다.

사용자는 실제 수행한 운동을 검색 및 선택할 수 있습니다.

예시:
- 벤치프레스
- 데드리프트
- 스쿼트
- 러닝

운동 검색 기능은 향후 자동완성 및 사용자 운동 추천 기능으로 확장될 수 있도록 고려되었습니다.

---

#### 3-3. Logging

운동 데이터를 입력하는 상태입니다.

세부 상태는 다음과 같이 구성됩니다.

- InputWeight
- InputSet

사용자는 운동 무게, 세트 수, 반복 횟수 등의 데이터를 입력하게 됩니다.

이 구조는 운동 종류별 입력 폼 분리 및 단계별 UX 최적화를 고려한 설계입니다.

---

#### 3-4. Calculating

운동 기록 저장 이후 보상 계산이 수행되는 상태입니다.

해당 상태는 내부적으로 다음 흐름으로 구성됩니다.

- EXP Update
- Tier Check
- Title Check

즉, 운동 저장 직후 다음 작업이 자동 수행됩니다.

- EXP Update
운동 수행량 기반 경험치 계산

- Tier Check
누적 경험치를 기반으로 티어 승급 여부 판별

- Title Check
특정 업적 및 조건 달성 여부 검사

이러한 구조는 단순 기록 기능을 게임화(Gamification) 시스템과 연결하기 위한 핵심 보상 메커니즘입니다.

모든 계산이 완료되면 사용자는 다시 메인 대시보드로 복귀합니다.

```text
Calculating -up-> Main : Update & Return
```

### 4. 마이페이지 및 칭호 관리 섹션 (Management Section)

마이페이지 영역은 사용자의 활동 기록과 개인화 요소를 관리하는 기능 영역입니다.


# 2) Verify State Diagram
![verify](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/Verifystate.png)

## Description  
이메일 인증 상태도는 Ex-Log 시스템에서 회원가입 과정 중 수행되는 이메일 인증 절차를 상태(State) 기반 흐름으로 표현한 다이어그램입니다.

본 상태도는 사용자가 이메일 인증을 요청한 순간부터 인증 완료 상태에 도달하기까지의 전체 과정을 단계별 상태 전이(State Transition) 형태로 정의하고 있으며, 이메일 인증 과정에서 발생할 수 있는 성공, 실패, 만료 상황까지 모두 포함하여 설계되었습니다.

이메일 인증 기능은 단순 회원가입 절차를 넘어, 사용자 신뢰성 확보와 계정 보안 강화를 위한 핵심 인증 시스템으로 동작합니다.  
특히 이메일 기반 인증은 허위 가입 방지, 계정 중복 생성 제한, 비밀번호 복구 기능 지원, 사용자 알림 시스템 연동 등의 기반 기능으로 활용됩니다.

시스템의 초기 상태는 `InputEmail` 상태입니다.

```text
[*] --> InputEmail
```

사용자는 회원가입 화면에서 인증받을 이메일 주소를 입력하게 되며, 시스템은 해당 상태에서 이메일 형식 유효성 검사 및 중복 여부 확인을 수행할 수 있습니다.

---

### 1. InputEmail

```text
state InputEmail
InputEmail : do/이메일 입력
```

`InputEmail` 상태는 사용자가 이메일 주소를 입력하는 단계입니다.

시스템은 다음과 같은 검증을 수행할 수 있습니다.

- 이메일 형식 검사
- 이미 가입된 이메일 여부 확인
- 공백 및 비정상 문자열 검증
- 허용 도메인 검사

사용자가 인증 요청 버튼을 클릭하면 시스템은 인증번호 발송 상태로 이동합니다.

```text
InputEmail -right-> SendCode : 인증 요청
```

---

### 2. SendCode

```text
state SendCode
SendCode : do/인증번호 발송
```

`SendCode` 상태에서는 시스템이 랜덤 인증번호를 생성하고 사용자의 이메일 주소로 전송합니다.

이 단계에서는 다음 작업이 수행됩니다.

- 인증번호 생성
- Redis 또는 메모리 저장소에 인증번호 저장
- 만료 시간 설정
- SMTP 서버 연결
- 이메일 발송

예를 들어 시스템은 다음과 같은 형태로 인증번호를 생성할 수 있습니다.

```text
[483921]
```

이후 `JavaMailSender` 또는 외부 메일 서버 API를 이용하여 사용자 이메일로 인증번호를 발송합니다.

메일 발송 성공 시 시스템은 인증번호 입력 대기 상태로 이동합니다.

```text
SendCode -right-> WaitCode : 발송 성공
```

반면 메일 서버 오류, 네트워크 장애, 잘못된 이메일 주소 등의 이유로 발송에 실패할 경우 다시 이메일 입력 상태로 복귀합니다.

```text
SendCode -left-> InputEmail : 발송 실패
```

이를 통해 사용자는 이메일 주소를 수정하거나 재시도를 수행할 수 있습니다.

---

### 3. WaitCode

```text
state WaitCode
WaitCode : do/인증번호 입력 대기
```

`WaitCode` 상태는 사용자가 이메일로 전달받은 인증번호를 입력하는 단계입니다.

시스템은 이 상태에서 다음 정보를 유지합니다.

- 인증번호 생성 시간
- 만료 시간
- 입력 시도 횟수
- 사용자 이메일 정보

사용자가 인증번호를 입력하면 시스템은 인증 검증 단계로 이동합니다.

```text
WaitCode -right-> VerifyCode : 인증번호 입력
```

또한 사용자가 제한 시간 내 인증을 완료하지 못할 경우 인증 만료 상태로 전이됩니다.

```text
WaitCode -down-> Expired : 제한 시간 초과
```

일반적으로 인증번호 유효 시간은 다음과 같이 설정됩니다.

- 3분
- 5분
- 10분

이는 보안성과 사용자 편의성을 고려한 정책 기반으로 설정됩니다.

---

### 4. VerifyCode

```text
state VerifyCode
VerifyCode : do/인증번호 검증
```

`VerifyCode` 상태에서는 사용자가 입력한 인증번호와 시스템에 저장된 인증번호를 비교 검증합니다.

시스템은 다음 조건들을 검사합니다.

- 인증번호 일치 여부
- 인증번호 만료 여부
- 이미 사용된 인증번호인지 여부
- 허용 입력 횟수 초과 여부

검증 성공 시 시스템은 인증 완료 상태로 이동합니다.

```text
VerifyCode -down-> Verified : 인증 성공
```

반면 인증번호가 일치하지 않을 경우 다시 입력 대기 상태로 복귀합니다.

```text
VerifyCode -up-> WaitCode : 인증 실패
```

이를 통해 사용자는 제한 횟수 내에서 인증번호를 다시 입력할 수 있습니다.

추가적으로 실제 서비스에서는 다음과 같은 보안 정책이 적용될 수 있습니다.

- 입력 횟수 제한
- 일정 횟수 실패 시 재발송 제한
- IP 기반 요청 제한
- 비정상 반복 요청 차단

이는 인증번호 brute-force 공격을 방지하기 위한 목적입니다.

---

### 5. Expired

```text
state Expired
Expired : do/인증시간 만료
```

`Expired` 상태는 인증번호 유효 시간이 초과된 상태를 의미합니다.

이 상태에서는 기존 인증번호가 무효화되며, 사용자는 더 이상 이전 인증번호를 사용할 수 없습니다.

사용자는 인증번호 재전송을 요청하여 다시 이메일 입력 상태로 이동할 수 있습니다.

```text
Expired -left-> InputEmail : 재전송
```

이를 통해 새로운 인증번호 기반 인증 절차가 다시 시작됩니다.

인증 만료 기능은 다음 목적을 가집니다.

- 보안 강화
- 인증번호 재사용 방지
- 장시간 노출 위험 감소
- 무단 인증 시도 제한

---

### 6. Verified

```text
state Verified
Verified : do/인증 완료 상태 저장
```

`Verified` 상태는 이메일 인증이 성공적으로 완료된 상태입니다.

시스템은 해당 상태에서 다음 작업을 수행할 수 있습니다.

- 인증 완료 여부 저장
- 회원가입 진행 가능 상태 변경
- 임시 인증 토큰 발급
- 세션 인증 상태 저장

인증이 완료되면 사용자는 회원가입 가능 상태로 전환됩니다.

```text
Verified --> [*] : 회원가입 가능 상태
```

즉, 이메일 인증 절차가 정상적으로 종료되었음을 의미합니다.

---

### 2. 보안 강화 구조

### 4. 확장 가능한 인증 시스템

# 3) Exercise State Diagram

## Description  
운동 기록 및 보상 시스템 상태도는 Ex-Log 서비스에서 사용자가 운동을 기록하는 순간부터 경험치, 티어, 칭호 등의 보상 처리가 완료되기까지의 전체 흐름을 상태(State) 기반으로 표현한 다이어그램입니다.

본 상태도는 단순 운동 기록 기능을 넘어, 운동 활동을 게임화(Gamification) 요소와 연결하여 사용자 참여도와 지속적인 운동 습관 형성을 유도하기 위한 시스템 구조를 나타냅니다.

특히 운동 기록 과정과 보상 처리 과정을 분리하여 표현함으로써, 사용자 입력 흐름과 서버 내부 보상 계산 로직을 명확하게 구분하고 있습니다.

전체 흐름은 크게 다음 두 영역으로 구성됩니다.

- 운동 기록 흐름 (ExerciseFlow)
- 보상 처리 시스템 (Reward)

시스템은 기본적으로 메인 대시보드 상태인 `Idle` 상태에서 시작됩니다.

```text
[*] --> Idle
```

---

### 1. Idle 상태

```text
state Idle
Idle : do/메인 대시보드 표시
```

`Idle` 상태는 사용자가 메인 화면에 위치한 기본 대기 상태를 의미합니다.

해당 상태에서는 다음과 같은 정보가 사용자에게 표시됩니다.

- 현재 티어
- 경험치
- 운동 잔디
- 최근 운동 기록
- 대표 칭호
- 스트릭 현황

즉, 사용자의 운동 활동 요약 정보가 표시되는 메인 허브 상태입니다.

사용자가 운동 시작 버튼을 클릭하면 운동 기록 흐름으로 이동합니다.

```text
Idle -right-> SelectCategory : 운동 시작
```

---

### 2. 운동 기록 흐름 (ExerciseFlow)

운동 기록 흐름은 사용자가 실제 운동 데이터를 입력하고 저장하는 과정을 담당합니다.

---

#### 2-1. SelectCategory

```text
state SelectCategory
SelectCategory : do/운동 카테고리 선택
```

사용자는 먼저 운동 카테고리를 선택합니다.

예시:
- 가슴
- 등
- 하체
- 어깨
- 유산소

해당 단계에서는 운동 목록 필터링 기능이 수행되며, 사용자가 원하는 운동 그룹을 빠르게 탐색할 수 있도록 지원합니다.

카테고리를 선택하면 운동 종목 선택 단계로 이동합니다.

```text
SelectCategory -right-> SelectExercise : 카테고리 선택
```

---

#### 2-2. SelectExercise

```text
state SelectExercise
SelectExercise : do/운동 종목 선택
```

사용자는 선택한 카테고리 내부에서 실제 수행한 운동 종목을 선택합니다.

예시:
- 벤치프레스
- 데드리프트
- 스쿼트
- 러닝

이 단계에서는 운동 검색 및 자동완성 기능이 함께 제공될 수 있으며, 사용자 운동 패턴 기반 추천 기능으로 확장 가능합니다.

운동 종목 선택 후에는 운동 데이터 입력 상태로 이동합니다.

```text
SelectExercise -right-> InputExerciseData : 종목 선택
```

---

#### 2-3. InputExerciseData

```text
state InputExerciseData
InputExerciseData : do/세트 수, 횟수,
거리 및 시간 입력
```

사용자는 실제 수행한 운동 데이터를 입력합니다.

입력 정보는 운동 종류에 따라 달라질 수 있습니다.

예시:
- 세트 수(Set)
- 반복 횟수(Reps)
- 운동 무게(Weight)
- 운동 거리(Distance)
- 운동 시간(Time)

예를 들어:
- 웨이트 운동 → 무게, 세트 수, 반복 횟수
- 러닝 → 거리, 시간
- 유산소 → 운동 지속 시간

사용자가 저장 버튼을 클릭하면 운동 기록 저장 상태로 이동합니다.

```text
InputExerciseData -down-> SaveExerciseLog : 저장 버튼 클릭
```

---

#### 2-4. SaveExerciseLog

```text
state SaveExerciseLog
SaveExerciseLog : do/운동 기록 저장
```

해당 상태에서는 사용자가 입력한 운동 데이터를 데이터베이스에 저장합니다.

시스템은 다음 작업을 수행할 수 있습니다.

- 운동 기록 저장
- 사용자 운동 통계 갱신
- 오늘 운동 여부 체크
- 운동 스트릭 업데이트
- 활동 로그 저장

운동 기록 저장이 성공적으로 완료되면 보상 처리 시스템으로 이동합니다.

```text
SaveExerciseLog -down-> GrassUpdate : 저장 성공
```

즉, 운동 기록 이벤트가 보상 시스템 실행의 트리거 역할을 수행합니다.

---

### 3. 보상 처리 시스템 (Reward)

보상 처리 시스템은 운동 기록 이후 수행되는 자동 성장 및 보상 로직을 담당합니다.

사용자의 운동 활동은 단순 저장에서 끝나는 것이 아니라 다음 요소들과 연결됩니다.

- 잔디 시스템
- 경험치 시스템
- 티어 시스템
- 칭호 시스템

---

#### 3-1. GrassUpdate

```text
state GrassUpdate
GrassUpdate : do/잔디 업데이트
```

운동 기록 이후 가장 먼저 수행되는 단계입니다.

시스템은 사용자의 운동 활동을 잔디(Contribution Graph)에 반영합니다.

이는 GitHub 잔디 시스템과 유사한 개념으로, 사용자의 운동 지속 여부를 시각적으로 표현합니다.

잔디 시스템의 목적:
- 운동 습관 형성 유도
- 사용자 성취감 강화
- 연속 운동 동기 부여

잔디 업데이트 이후 경험치 계산 단계로 이동합니다.

```text
GrassUpdate --> ExpUpdate
```

---

#### 3-2. ExpUpdate

```text
state ExpUpdate
ExpUpdate : do/경험치 계산
```

운동 기록 기반 경험치 계산 단계입니다.

시스템은 다음 데이터를 기반으로 경험치를 산정할 수 있습니다.

- 운동 시간
- 운동 강도
- 운동 종류
- 세트 수
- 수행 빈도

예시:
- 기본 운동 경험치 지급
- 고강도 운동 추가 보상
- 장시간 운동 보너스

이후 시스템은 사용자의 운동량에 따라 분기 처리됩니다.

---

#### 일반 경험치 처리

```text
ExpUpdate --> TierCheck : 일반 경험치 처리
```

기본 경험치만 지급된 경우 바로 티어 검사 단계로 이동합니다.

---

#### 추가 경험치 지급

```text
ExpUpdate --> BonusExp : 3개 이상 운동 완료
```

하루 동안 3개 이상의 운동 종목을 완료한 경우 추가 경험치 지급 상태로 이동합니다.

이는 사용자의 다양한 운동 참여를 유도하기 위한 보상 정책입니다.

---

#### 3-3. BonusExp

```text
state BonusExp
BonusExp : do/3종목 완료
추가 경험치 지급
```

해당 상태에서는 다중 운동 수행 보너스 경험치를 지급합니다.

예시:
- +50 EXP 추가 지급
- 특별 보너스 메시지 출력

이 기능은 특정 운동만 반복 수행하는 패턴을 줄이고 다양한 운동 참여를 장려하기 위한 목적을 가집니다.

보너스 경험치 지급 후 티어 확인 단계로 이동합니다.

```text
BonusExp --> TierCheck
```

---

#### 3-4. TierCheck

```text
state TierCheck
TierCheck : do/티어 조건 확인
```

누적 경험치를 기반으로 사용자 티어 승급 여부를 확인합니다.

예시 티어:
- Bronze
- Silver
- Gold
- Platinum

시스템은 내부 조건을 기반으로 승급 가능 여부를 검사합니다.

---

### 티어 승급 성공

```text
TierCheck --> TierUp : 티어 조건 만족
```

조건을 만족하면 티어 상승 처리 상태로 이동합니다.

---

### 티어 유지

```text
TierCheck --> TitleCheck : 티어 유지
```

조건을 만족하지 못하면 기존 티어를 유지한 채 칭호 검사 단계로 이동합니다.

---

#### 3-5. TierUp

```text
state TierUp
TierUp : do/티어 상승 처리
```

사용자의 티어를 상위 단계로 변경합니다.

이 단계에서는 다음 작업이 수행될 수 있습니다.

- 티어 데이터 갱신
- 승급 알림 출력
- 프로필 반영
- 랭킹 시스템 갱신

티어 상승 이후에는 칭호 검사 단계로 이동합니다.

```text
TierUp --> TitleCheck
```

---

#### 3-6. TitleCheck

```text
state TitleCheck
TitleCheck : do/칭호 달성 여부 확인
```

시스템은 사용자의 운동 활동 데이터를 기반으로 새로운 칭호 획득 가능 여부를 확인합니다.

검사 조건 예시:
- 누적 운동 횟수
- 특정 운동 달성
- 연속 운동 일수
- 특정 티어 달성

예시 칭호:
- 초보 헬스인
- 러닝 마스터
- 꾸준한 운동러
- 근성의 지배자

---

#### 칭호 획득 성공

```text
TitleCheck --> TitleGranted : 칭호 조건 달성
```

새로운 칭호 지급 상태로 이동합니다.

---

#### 칭호 없음

```text
TitleCheck -left-> Idle : 칭호 없음
```

획득 가능한 칭호가 없을 경우 메인 상태로 복귀합니다.

---

#### 3-7. TitleGranted

```text
state TitleGranted
TitleGranted : do/새로운 칭호 지급
```

새로운 칭호를 사용자에게 지급하는 단계입니다.

이 단계에서는 다음 작업이 수행될 수 있습니다.

- 칭호 DB 저장
- 획득 알림 출력
- 대표 칭호 설정 가능 상태 변경
- 프로필 반영

보상 처리가 모두 완료되면 시스템은 다시 Idle 상태로 복귀합니다.

```text
TitleGranted -left-> Idle : 처리 완료
```

---

#### 시스템 구조적 특징

본 상태도는 단순 운동 기록 기능이 아니라, 운동 활동과 사용자 성장 시스템을 결합한 게임화 기반 구조를 표현합니다.

특히 다음 특징을 가집니다.

---

### 1. 운동 기록과 보상 시스템 분리

운동 입력 흐름과 내부 보상 계산 로직을 분리하여 유지보수성과 확장성을 향상시켰습니다.

---

### 2. 게임화(Gamification) 구조

운동 → 경험치 → 티어 → 칭호로 이어지는 성장 흐름을 통해 사용자 지속 참여를 유도합니다.

---

### 3. 시각적 성취 시스템 제공

잔디, 티어, 칭호 시스템을 통해 사용자의 성취를 시각적으로 표현합니다.

---

### 4. 확장 가능한 구조

현재 상태 구조는 다음 기능으로 확장 가능합니다.

- 운동 업적 시스템
- 일일 미션
- 친구 경쟁 기능
- 랭킹 시스템
- 시즌제 티어
- 희귀 칭호 시스템
- AI 운동 추천

이를 통해 단순 운동 기록 앱이 아닌, 장기적인 운동 습관 형성과 사용자 몰입을 지원하는 성장형 플랫폼으로 발전할 수 있도록 설계되었습니다.

# 4) Mypage State Diagram
![Mypage](https://github.com/KDWorld81/Ex-Log_Docs/blob/main/chap3Images/Mypagestate.png)


## Description  
마이페이지 및 칭호 관리 상태도는 Ex-Log 시스템에서 사용자가 자신의 운동 활동 정보와 칭호 데이터를 조회하고 관리하는 전체 흐름을 상태(State) 기반으로 표현한 다이어그램입니다.

본 상태도는 단순 프로필 조회 기능을 넘어, 사용자의 운동 성과와 업적을 시각적으로 표현하고 관리할 수 있는 개인화 시스템 구조를 나타냅니다.  
특히 운동 기록 조회, 칭호 도감, 대표 칭호 설정 기능 등을 하나의 통합된 흐름으로 구성하여 사용자 정체성과 성취 표현 요소를 강화하도록 설계되었습니다.

전체 흐름은 크게 다음 영역으로 구성됩니다.

- 메인 대시보드(MainDashboard)
- 마이페이지(MyPage)
- 운동 기록 조회(HistoryView)
- 칭호 도감(TitleEncyclopedia)
- 대표 칭호 설정(MainTitleSetting)

시스템의 초기 상태는 `MainDashboard` 상태입니다.

```text
[*] --> MainDashboard
```

---

### 1. MainDashboard

```text
state MainDashboard
MainDashboard : do/메인 화면 표시
```

`MainDashboard` 상태는 사용자가 로그인 이후 진입하는 메인 화면 상태입니다.

해당 화면에서는 다음과 같은 핵심 정보가 사용자에게 제공됩니다.

- 현재 티어
- 경험치
- 운동 잔디
- 최근 운동 기록
- 대표 칭호
- 스트릭 정보

즉, 사용자의 전체 운동 활동 요약 정보를 확인할 수 있는 메인 허브 역할을 수행합니다.

사용자가 마이페이지 버튼 또는 프로필 영역을 클릭하면 마이페이지 상태로 이동합니다.

```text
MainDashboard -down-> MyPage : 마이페이지 이동
```

---

### 2. MyPage

```text
state MyPage
MyPage : do/프로필 및 활동 정보 표시
```

`MyPage` 상태는 사용자 개인 프로필과 활동 정보를 조회하는 중심 상태입니다.

해당 화면에서는 사용자의 운동 성과와 활동 내역이 종합적으로 표시됩니다.

포함 정보 예시:
- 프로필 이미지
- 닉네임
- 대표 칭호
- 현재 티어
- 누적 경험치
- 운동 스트릭
- 총 운동 횟수
- 최근 운동 활동

즉, 사용자의 운동 이력을 시각적으로 요약하여 보여주는 개인화 프로필 허브입니다.

마이페이지 상태에서는 다음 기능으로 이동할 수 있습니다.

- 운동 기록 조회
- 칭호 도감 조회
- 대표 칭호 설정

---

### 3. 운동 기록 조회 (HistoryView)

```text
state HistoryView
HistoryView : do/운동 기록 조회
```

`HistoryView` 상태에서는 사용자의 운동 기록 데이터를 조회할 수 있습니다.

시스템은 저장된 운동 데이터를 기반으로 다음 정보를 제공합니다.

- 날짜별 운동 기록
- 운동 종목
- 세트 수 및 반복 횟수
- 운동 시간 및 거리
- 운동 수행 통계

운동 기록은 일반적으로 페이지네이션(Paging) 기반으로 조회되며, 많은 데이터도 효율적으로 탐색할 수 있도록 설계됩니다.

향후 다음 기능으로 확장 가능합니다.

- 날짜 필터링
- 운동 종류 검색
- 운동 통계 그래프
- 주간/월간 분석
- 운동 비교 기능

사용자가 뒤로가기 버튼을 선택하면 다시 마이페이지 상태로 복귀합니다.

```text
HistoryView -left-> MyPage : 뒤로가기
```

---

### 4. 칭호 도감 (TitleEncyclopedia)

```text
state TitleEncyclopedia
```

`TitleEncyclopedia` 상태는 사용자가 획득한 칭호 및 미획득 칭호를 조회할 수 있는 칭호 관리 시스템입니다.

이 기능은 단순 보상 저장 기능이 아니라, 사용자 수집 욕구와 지속적인 운동 동기 부여를 강화하기 위한 게임화(Gamification) 요소로 설계되었습니다.

칭호 도감은 내부적으로 다음 두 상태로 구성됩니다.

- TitleList
- TitleDetail

---

#### 4-1. TitleList

```text
state TitleList
TitleList : do/획득 및 미획득
칭호 목록 표시
```

`TitleList` 상태에서는 전체 칭호 목록이 사용자에게 표시됩니다.

칭호는 다음과 같이 구분될 수 있습니다.

- 획득 완료 칭호
- 미획득 칭호
- 숨겨진 희귀 칭호
- 이벤트 한정 칭호

사용자는 각 칭호의 획득 여부를 확인할 수 있으며, 자신의 수집 현황도 함께 확인할 수 있습니다.

예시 칭호:
- 운동 입문자
- 꾸준한 운동러
- 러닝 마스터
- 근성의 지배자

칭호 시스템은 사용자의 장기적인 운동 참여를 유도하기 위한 핵심 보상 요소입니다.

사용자가 특정 칭호를 선택하면 상세 조회 상태로 이동합니다.

```text
TitleList -right-> TitleDetail : 칭호 선택
```

---

#### 4-2. TitleDetail

```text
state TitleDetail
TitleDetail : do/칭호 조건 상세 조회
```

`TitleDetail` 상태에서는 특정 칭호의 상세 정보를 조회할 수 있습니다.

표시 정보 예시:
- 칭호 이름
- 칭호 설명
- 획득 조건
- 현재 달성 진행도
- 획득 여부

예시:
```text
칭호: 꾸준한 운동러
조건: 30일 연속 운동
진행도: 18 / 30
```

이를 통해 사용자는 자신의 목표를 명확하게 인지할 수 있으며, 운동 지속 동기를 강화할 수 있습니다.

---

### 5. 대표 칭호 설정 (MainTitleSetting)

```text
state MainTitleSetting
```

대표 칭호 설정 기능은 사용자가 자신이 보유한 칭호 중 하나를 프로필 대표 칭호로 지정하는 기능입니다.

이는 단순한 업적 저장이 아니라, 사용자 정체성과 성취 표현 요소를 강화하기 위한 개인화 기능입니다.

대표 칭호 설정 흐름은 다음 상태로 구성됩니다.

- OwnershipCheck
- UpdateMainTitle

---

#### 5-1. OwnershipCheck

```text
state OwnershipCheck
OwnershipCheck : do/칭호 보유 여부 확인
```

사용자가 대표 칭호 설정 요청을 보내면 시스템은 먼저 해당 칭호의 실제 보유 여부를 검증합니다.

검증 목적:
- 미획득 칭호 설정 방지
- 임의 요청 변조 차단
- 데이터 무결성 유지

예를 들어 사용자가 직접 존재하지 않는 칭호 ID를 요청하더라도, 서버 측 검증을 통해 차단됩니다.

이는 보안성과 데이터 신뢰성을 유지하기 위한 핵심 검증 단계입니다.

검증이 완료되면 대표 칭호 갱신 단계로 이동합니다.

```text
OwnershipCheck --> UpdateMainTitle
```

---

#### 5-2. UpdateMainTitle

```text
state UpdateMainTitle
UpdateMainTitle : do/대표 칭호 변경
```

검증이 완료되면 시스템은 사용자의 대표 칭호 정보를 데이터베이스에 갱신합니다.

이 단계에서는 다음 작업이 수행될 수 있습니다.

- 대표 칭호 DB 저장
- 프로필 정보 갱신
- 캐시 데이터 업데이트
- 사용자 UI 반영

변경 완료 후 사용자는 다시 마이페이지 상태로 복귀합니다.

```text
UpdateMainTitle -up-> MyPage : 변경 완료
```

이후 대표 칭호는 사용자 프로필 및 커뮤니티 영역에 즉시 반영됩니다.

---

### 6. 마이페이지 종료

사용자가 마이페이지를 종료하면 시스템은 다시 메인 대시보드 상태로 복귀합니다.

```text
MyPage -up-> MainDashboard : 종료
```

이를 통해 사용자는 언제든 메인 서비스 흐름으로 자연스럽게 돌아갈 수 있습니다.

# 8. Design Considerations

### 1) Performance Optimization using Redis

본 시스템은 사용자 운동 기록 저장, 경험치 계산, 칭호 부여와 같이 반복적인 데이터 조회가 빈번하게 발생하는 서비스라는 점을 고려하여 Redis를 활용한 캐시 구조를 적용하도록 설계하였습니다.

기존의 관계형 데이터베이스(MySQL)만을 사용할 경우, 동일한 데이터에 대한 반복 조회 요청이 많아질수록 데이터베이스 부하가 증가할 수 있습니다. 이러한 문제를 해결하기 위해 In-Memory 기반 데이터 저장소인 Redis를 도입하여 자주 조회되는 데이터를 캐싱하고, 데이터베이스 접근 횟수를 최소화하는 방향으로 설계하였습니다.

또한 Redis의 빠른 읽기 및 쓰기 성능을 활용하여 향후 실시간 운동 랭킹 시스템, 사용자 활동 통계 기능, 최근 운동 기록 조회 기능 등 다양한 확장 기능에도 활용할 수 있도록 고려하였습니다. 이를 통해 전체적인 서비스 응답 속도를 향상시키고, 대규모 사용자 요청에도 안정적으로 대응할 수 있는 구조를 목표로 설계하였습니다.

---

### 2) Security-Oriented Authentication Design using JWT

보안 측면에서는 JWT(Json Web Token) 기반 인증 방식을 적용하여 Stateless 구조로 인증 시스템을 설계하였습니다.

기존 세션(Session) 기반 인증 방식은 서버 측에서 사용자 인증 정보를 직접 저장하고 관리해야 하기 때문에 서버 부하가 증가할 수 있으며, 확장성 측면에서도 한계가 존재할 수 있습니다. 이를 보완하기 위해 JWT 기반 인증 방식을 도입하여 서버의 세션 저장 부담을 줄이고, 보다 효율적인 인증 처리가 가능하도록 설계하였습니다.

사용자는 로그인 이후 Access Token과 Refresh Token을 발급받게 되며, Access Token을 통해 API 요청에 대한 인증을 수행하고, Refresh Token을 활용하여 새로운 Access Token을 재발급받을 수 있도록 구성하였습니다.

또한 Spring Security와 연계하여 인증(Authentication) 및 인가(Authorization)를 체계적으로 처리할 수 있도록 설계하였으며, 사용자 권한에 따른 API 접근 제어가 가능하도록 고려하였습니다.

특히 Refresh Token은 별도의 RefreshToken 엔티티를 통해 관리하도록 설계하였습니다. 이를 통해 사용자별 토큰 정보를 데이터베이스에서 체계적으로 관리할 수 있도록 하였으며, 로그아웃 시 토큰 무효화 처리 및 토큰 재발급 과정을 안정적으로 수행할 수 있도록 고려하였습니다.

이를 통해 사용자 인증 과정의 안정성과 보안성을 동시에 확보하는 방향으로 시스템을 구성하였습니다.

---

### 3) Container-Based Deployment using Docker & Docker Compose

프로젝트 실행 환경의 일관성과 배포 편의성을 높이기 위해 Docker 기반 컨테이너 환경을 적용하도록 설계하였습니다.

개발 환경과 운영 환경 간 설정 차이로 인해 발생할 수 있는 문제를 최소화하기 위하여 Spring Boot 애플리케이션, MySQL, Redis 등의 서비스를 각각 독립적인 컨테이너로 구성하였습니다. 이를 통해 실행 환경이 달라지더라도 동일한 환경에서 안정적으로 애플리케이션을 실행할 수 있도록 고려하였습니다.

또한 여러 컨테이너를 효율적으로 관리하기 위해 Docker Compose를 활용하여 전체 서비스를 통합적으로 실행 및 관리할 수 있도록 설계하였습니다. Docker Compose를 사용함으로써 하나의 설정 파일만으로 여러 서비스를 동시에 실행할 수 있으며, 개발 및 배포 과정의 복잡성을 줄일 수 있도록 고려하였습니다.

이를 통해 팀원 간 동일한 개발 환경을 쉽게 공유할 수 있도록 하였으며, 새로운 개발자가 프로젝트에 참여하더라도 빠르게 개발 환경을 구성할 수 있도록 설계하였습니다.

또한 컨테이너 기반 구조를 통해 서비스 확장 및 유지보수가 용이하도록 고려하였으며, 향후 마이크로서비스 구조로 확장할 수 있는 가능성도 고려하여 설계하였습니다.

---

### 4) AWS EC2 Deployment & CI/CD Pipeline

배포 환경은 AWS EC2 서버를 기반으로 구성하며, Docker Compose를 활용하여 서버에서 애플리케이션을 실행하도록 설계하였습니다.

애플리케이션 실행에 필요한 Spring Boot, MySQL, Redis 환경을 Docker Container 형태로 구성함으로써 서버 환경에 대한 의존성을 최소화하고, 안정적인 배포가 가능하도록 고려하였습니다.

또한 유지보수 효율성과 배포 자동화를 위하여 GitHub Actions 기반 CI/CD 파이프라인 구축을 고려하였습니다.

CI/CD 환경에서는 개발자가 GitHub Repository에 코드를 Push하면 자동으로 Build가 수행되며, 이후 Docker Image 생성 및 EC2 서버 배포까지 자동으로 진행될 수 있도록 설계하였습니다. 이를 통해 수동 배포 과정에서 발생할 수 있는 설정 오류나 배포 실수를 줄이고, 보다 안정적인 서비스 운영이 가능하도록 고려하였습니다.

특히 Docker Compose를 활용하여 서버 내 여러 컨테이너를 통합 관리할 수 있도록 구성함으로써 서비스 재시작 및 버전 업데이트를 보다 효율적으로 수행할 수 있도록 설계하였습니다.

또한 자동화된 배포 환경을 구축함으로써 기능 수정 및 업데이트 사항을 빠르게 반영할 수 있도록 하였으며, 장기적인 유지보수 측면에서도 효율적인 프로젝트 운영이 가능하도록 고려하였습니다.
